# AI 추론 엔진 마이그레이션 발표 대본

> **발표 제목**: OnSafe 낙상 감지 AI 서버 — 추론 엔진 전면 교체 작업 설명  
> **대상**: 팀원 / 협업자  
> **소요 시간**: 약 20~30분

---

## 🎤 오프닝

안녕하세요.  
오늘은 OnSafe 낙상 감지 서비스의 AI 추론 엔진을 어떻게 교체했는지,  
그리고 왜 그 과정이 필요했는지를 단계별로 설명드리겠습니다.

---

## 1. 배경 — "왜 바꿔야 했나?"

### 기존 구조의 문제

처음 구현할 때는 이런 방식이었습니다.

```
Android 폰 → JPEG 사진 → 서버(MediaPipe 실행) → 추론
```

Android가 카메라로 찍은 사진을 그대로 서버로 보내면,  
서버에서 MediaPipe라는 라이브러리로 사람의 관절 좌표를 뽑고,  
그 좌표로 "낙상 위험이 몇 %인지" 계산하는 구조였습니다.

이 방식에는 세 가지 큰 문제가 있었습니다.

**첫째, 비용 문제**

JPEG 한 장이 약 20KB입니다.  
초당 30프레임을 서버로 보내면 1초에 600KB, 1시간에 2GB가 넘습니다.  
사용자가 늘어날수록 네트워크 비용이 폭발적으로 증가합니다.

**둘째, 모델이 구형**

기존 모델은 Decision Tree였습니다.  
원본 AI 서버(`OnSafe/ai-server`)는 이미 XGBoost로 업그레이드되어 있었는데,  
백엔드 서버(`On-safe-backend`)는 아직 구형 모델을 쓰고 있었습니다.  
두 레포 간 불일치 상태였습니다.

**셋째, 전처리가 부실**

기존 코드는 프레임 한 장만 보고 판단했습니다.  
사람이 잠깐 비틀거려도 낙상으로 오판하거나,  
실제 낙상인데 한 프레임만 봐서 놓치는 경우가 생길 수 있었습니다.

---

### 목표 구조

그래서 이렇게 바꾸기로 했습니다.

```
Android 폰 (on-device MediaPipe) → 관절 좌표 JSON → 서버(XGBoost 추론)
```

Android가 직접 MediaPipe로 관절 좌표를 뽑고,  
좌표값(JSON, 약 2KB)만 서버로 보냅니다.  
서버는 좌표를 받아서 XGBoost로 추론만 합니다.

트래픽이 1/10로 줄고, 서버 부하도 크게 감소합니다.

이 작업을 6단계(Step)로 나눠서 진행했습니다.

---

## 2. Step 1 — 학습 모델 파일 교체

### pkl 파일이란?

먼저 `.pkl` 파일이 뭔지 간단히 설명하겠습니다.

Python에서 학습된 AI 모델을 파일로 저장할 때 쓰는 포맷입니다.  
우리 서버에는 두 가지 pkl 파일이 필요합니다.

- **`xgb_model.pkl`**: 학습된 XGBoost 모델 — "이 데이터면 낙상 확률이 몇 %인지" 판단하는 두뇌
- **`scaler.pkl`**: StandardScaler — 입력 데이터를 모델이 이해할 수 있는 숫자 범위로 변환하는 도구

### 교체 작업

```
삭제: decision_tree_model.pkl   ← 구형 모델
추가: xgb_model.pkl             ← 신형 XGBoost 모델
교체: scaler.pkl                ← 새 모델에 맞는 변환 도구
```

### ⚠️ 반드시 두 파일을 쌍으로 교체해야 합니다

왜냐하면 이 둘은 **같은 데이터로 함께 학습**되었기 때문입니다.

비유를 들면 이렇습니다.  
scaler는 "키 170cm를 0.5로 변환하는 기준표"이고,  
모델은 "0.5라는 값을 보고 학습된 판단기"입니다.  
기준표는 그대로인데 판단기만 바꾸면 —  
새 판단기는 다른 기준으로 학습됐으니 — 완전히 틀린 결과가 나옵니다.

## 3. Step 2 — 추론 엔진(engine.py) 전면 재작성

engine.py는 낙상 감지의 핵심입니다.  
"관절 좌표 데이터를 받아서 위험 점수를 뽑아주는" 역할을 합니다.

### 제거한 것들

| 제거 항목 | 이유 |
|-----------|------|
| `import cv2`, `import mediapipe` | Android가 직접 좌표를 뽑으니 서버에서 불필요 |
| JPEG 입력 함수 `infer_frame(jpeg_bytes)` | 좌표 입력 함수로 교체 |
| arccos 각도 계산 | 수치 불안정 → arctan2로 교체 |
| center 피처 6개 | 실제 필요한 2개만 유지 |

### 새로 추가한 전처리 파이프라인

좌표값을 그대로 모델에 넣는 게 아니라,  
데이터를 단계별로 가공한 뒤 모델에 입력합니다.  
이 과정을 Step 2~6이라고 부릅니다.

---

#### 전처리 Step 2 — NaN 보정

**문제**: 관절이 옷에 가려지거나 화면 밖으로 벗어나면 좌표값이 없습니다 (NaN).

**해결**:
1. MediaPipe 신뢰도가 0.3 미만인 좌표 → NaN으로 표시
2. 통계적 이상치(평균에서 3배 표준편차 이상) → NaN으로 표시
3. NaN은 앞뒤 프레임 값의 평균으로 보간

쉽게 말하면 — 손이 잠깐 가렸을 때 "직전 위치와 직후 위치의 중간 어딘가에 있었겠지"로 추정하는 것입니다.

---

#### 전처리 Step 3 — Savitzky-Golay 스무딩

**문제**: 카메라로 추출한 좌표는 프레임마다 미세하게 흔들립니다 (노이즈).

**해결**: 연속된 7프레임 구간의 데이터를 2차 곡선으로 수학적으로 매끄럽게 다듬습니다.

손을 떨면서 그린 선을 자연스러운 곡선으로 다듬는 것과 같습니다.  
이렇게 하면 실제 동작 패턴이 더 명확하게 드러납니다.

---

#### 전처리 Step 4 — 골반 정규화

**문제**: 사람마다 키가 다르고, 카메라 위치나 거리도 다릅니다.

**해결**:
1. 양쪽 골반의 중앙점을 원점(0, 0, 0)으로 이동
2. 양쪽 골반 간 거리를 기준값 1로 정규화

키 170cm인 사람과 160cm인 사람의 데이터를 같은 스케일로 맞추는 작업입니다.  
모델이 "이 사람이 크다 작다"가 아니라 "동작 자체"를 보게 합니다.

---

#### 전처리 Step 5 — 47개 피처 계산

이 단계가 가장 중요합니다.

좌표(x, y, z)를 그대로 쓰는 게 아니라 **"움직임의 특성"을 수치로 표현**합니다.

15개 관절 각각에 대해 3가지를 계산합니다.

| 피처 | 의미 | 낙상 시 특징 |
|------|------|-------------|
| 각도 | 관절이 얼마나 구부러졌나 | 골반·무릎이 급격히 꺾임 |
| 각속도 | 구부러지는 속도 | 갑자기 빨라짐 |
| 각가속도 | 속도의 변화율 | 급격한 가속 발생 |

15개 관절 × 3가지 = 45개  
무게중심 이동거리 + 무게중심 속도 = 2개  
합계 **47개 피처**

이 47개 숫자가 모델에 들어가는 최종 입력입니다.

---

#### 전처리 Step 6 — StandardScaler 정규화

**문제**: 피처마다 단위와 범위가 다릅니다.
- 각도: 0~180도
- 각속도: 0~수천 deg/s

이 상태로 모델에 넣으면 숫자가 큰 피처가 영향을 더 많이 줍니다.

**해결**: scaler.pkl이 모든 피처를 평균 0, 표준편차 1로 변환합니다.  
이렇게 하면 모델이 모든 피처를 공평하게 비교할 수 있습니다.

---

### 추론 주기 설계 — WINDOW_SIZE와 STRIDE

```python
WINDOW_SIZE = 30  # 30프레임이 쌓여야 추론 시작 (약 1초)
STRIDE      = 5   # 5프레임마다 한 번씩만 추론
```

**WINDOW_SIZE = 30인 이유**  
1장의 사진만 보면 오탐이 많습니다.  
약 1초(30프레임) 동안의 연속 동작을 봐야 "이게 진짜 넘어지는 건지" 판단할 수 있습니다.

**STRIDE = 5인 이유**  
30프레임이 쌓인 후에도 매 프레임마다 추론하면 서버 부하가 큽니다.  
5프레임마다 한 번씩만 추론해도 충분히 실시간성이 유지됩니다.

```
frame 0~28  → 데이터 부족, 추론 건너뜀
frame 29    → 첫 추론 실행 ← 29 % 5 == 4
frame 30~33 → 건너뜀
frame 34    → 두 번째 추론  ← 34 % 5 == 4
frame 35~38 → 건너뜀
frame 39    → 세 번째 추론  ← 39 % 5 == 4
```

---

### 위험 레벨 임계값 (2026-07-13 최신 — 최초 51/76에서 50/75로 재수정됨)

```python
WARNING_THRESHOLD  = 50.0  # 50점 초과 → "주의"
CRITICAL_THRESHOLD = 75.0  # 75점 초과 → "위험"
```

이 기준으로 Kotlin 서버의 RiskLevel.kt와 통일되어 있습니다.  
Python과 Kotlin이 같은 기준으로 레벨을 판단합니다.

> 발표 당시(PR #14, 이 문서 최초 작성)에는 51/76 · `>=` 비교였으나, 경계값(51.0/76.0)이
> 잘못된 등급으로 분류되는 문제가 발견되어 2026-06-05(Kotlin)·2026-06-24(Python)에
> 50/75 · `>` 비교로 재수정되었습니다.

---

### `_classify_level()` 추가 (PR #14, 2026-06-24 재수정)

```python
def _classify_level(score: float) -> str:
    if score > 75.0:
        return "위험"
    if score > 50.0:
        return "주의"
    return "정상"
```

처음에는 레벨 분류 로직이 service.py에 중복으로 있었습니다.  
이 함수를 engine.py 안으로 옮겨서 추론 결과에 level 필드를 포함시키고,  
service.py의 중복 코드는 제거했습니다.

## 4. Step 3 — buffer.py 수정

buffer.py는 Redis와 통신해서 점수 캐싱, 알림 쿨다운 등을 관리합니다.

### 핵심 변경 — "30프레임 판단 주체" 이전

기존 구조:
```
Redis 카운터(buffer.py)가 30 이상인지 체크
  → True면 engine에게 "추론해"라고 지시
```

현재 구조:
```
engine 내부 deque(메모리 버퍼)가 직접 30개 여부 체크
  → 30개 차면 알아서 추론 실행
```

왜 바꿨을까요? Redis 카운터와 메모리 버퍼가 불일치하는 상황이 생길 수 있기 때문입니다.

예를 들어 서버를 재시작하면 메모리 버퍼는 0으로 초기화되지만,  
Redis에 이전 카운터 값이 남아있을 수 있습니다.  
이 경우 Redis는 "30개 쌓였다"고 판단하지만 실제로는 0개인 상황이 발생합니다.

이 불일치를 없애기 위해 판단 주체를 engine 하나로 단일화했습니다.

## 5. Step 4 — service.py 수정

service.py는 추론 결과를 받아서 비즈니스 로직을 처리합니다.

### 피처 목록 불일치 수정

```python
# 수정 전 — engine이 만들지도 않는 피처를 저장하려 했음 (버그)
'center_acceleration', 'center_displacement', 'center_velocity_change' ...

# 수정 후 — engine이 실제로 만드는 피처와 일치
'center_distance', 'center_speed'
```

engine.py가 만드는 피처와 service.py가 저장하려는 피처 목록이 맞지 않았습니다.  
이를 정확히 맞춰서 수정했습니다.

## 6. Step 5 — WebSocket 프로토콜 전환

이 단계가 이번 마이그레이션에서 **가장 큰 변화**입니다.

### HTTP vs WebSocket

**기존 방식 (HTTP)**
```
Android → POST 요청 (JPEG 사진 첨부) → 서버 응답
         → POST 요청 (JPEG 사진 첨부) → 서버 응답
         → POST 요청 (JPEG 사진 첨부) → 서버 응답
         (매 프레임마다 연결 수립 → 데이터 전송 → 연결 종료 반복)
```

**현재 방식 (WebSocket)**
```
Android → 연결 수립 (1회)
        → frame 전송 → frame 전송 → frame 전송 ... (연결 유지)
        ← result    ← result    ← result    ...
        → 연결 종료 (1회)
```

WebSocket은 한 번 연결하면 끊지 않고 계속 데이터를 주고받습니다.  
HTTP처럼 매 프레임마다 연결을 맺고 끊는 오버헤드가 없습니다.

### 메시지 형식

**Android → 서버**
```json
// 1. 연결 초기화 (처음 한 번)
{"type": "init", "user_id": "user001", "device_id": "device001"}

// 2. 매 프레임 좌표 전송
{
  "type": "frame",
  "frame": 42,
  "timestamp": 2.8,
  "landmarks": [
    {"x": 0.5, "y": 0.3, "z": -0.1, "v": 0.95},
    ...  // 33개 관절 좌표
  ]
}
```

**서버 → Android**
```json
// 추론 결과 (30프레임마다)
{
  "type": "result",
  "fall_score": 82.5,
  "fall": true,
  "level": "위험"
}
```

---

### 이 전환으로 MediaPipe가 서버에서 제거된 이유

기존에는 Android가 JPEG를 보내면 서버가 MediaPipe로 관절을 찾아야 했습니다.  
이제는 Android가 이미 관절 좌표를 뽑아서 보내기 때문에  
서버는 그냥 숫자를 받아서 계산만 하면 됩니다.

MediaPipe가 서버에 있을 이유가 없어진 것입니다.

## 8. 전체 흐름 정리

지금까지 설명한 내용을 한 그림으로 보면 이렇습니다.

```
┌─────────────────────────────────────────────────────┐
│                   Android 폰                         │
│  카메라 → MediaPipe → 33개 관절 좌표 추출            │
└────────────────────┬────────────────────────────────┘
                     │ WebSocket (좌표 JSON, ~2KB/frame)
┌────────────────────▼────────────────────────────────┐
│               Python AI 서버                         │
│                                                      │
│  30프레임 버퍼에 쌓기                                │
│    ↓                                                 │
│  Step2: 가려진 관절 좌표 보간                        │
│  Step3: 좌표 떨림 제거 (스무딩)                      │
│  Step4: 키·위치 차이 제거 (골반 정규화)              │
│  Step5: 47개 특징값 계산 (각도/속도/가속도)          │
│  Step6: 단위 통일 (StandardScaler)                   │
│    ↓                                                 │
│  XGBoost → score(0~100), fall(bool), level           │
│                                                      │
│  score > 75 또는 fall=True → 위험·낙상 처리         │
│  score > 50 (75 이하) → 5분 쿨다운 확인 후 주의 처리 │
└────────────────────┬────────────────────────────────┘
                     │ HTTP (내부 API)
┌────────────────────▼────────────────────────────────┐
│              Kotlin Spring 서버                       │
│  Firestore 저장 + FCM 푸시 알림                      │
└────────────────────┬────────────────────────────────┘
                     │ FCM
```

## 9. 비용 분석 — 왜 이 구조로 바꿨는지 숫자로 보기

이 전환의 가장 큰 동기 중 하나가 비용입니다.

### 트래픽 비교 (사용자 1명, 1시간 기준)

| 방식 | 프레임당 크기 | 시간당 트래픽 |
|------|-------------|-------------|
| 기존 JPEG 30fps | 20KB | **2,160 MB/h** |
| 현재 landmark 30fps | 2KB | **216 MB/h** |
| 절감 | | **90% 감소** |

### 월 비용 비교 (사용자 500명 기준)

| 구조 | 월 비용 |
|------|---------|
| 기존 (JPEG 30fps 서버 경유) | **약 $11,889** |

| 현재 + 보호자 릴레이 2fps | **약 $925** |

비용의 90% 이상이 보호자 앱으로 영상을 전달하는 outbound 트래픽에서 발생합니다.  
AI 추론 서버 비용 자체는 월 $37(t3.medium) 수준으로 미미합니다.

현재 보호자 릴레이 방식(Option A vs B)은 팀에서 협의 중입니다.

## 10. 향후 고려사항 — Option C (On-device 추론)

마지막으로 미래에 고려할 수 있는 구조를 소개합니다.

서버 AI를 제거하고 Android 폰에서 직접 추론하는 방식입니다.

```
현재: Android → (좌표) → 서버 → (추론) → 결과
Option C: Android → (좌표 추출 + 추론 모두 기기에서 직접) → 결과
```

**장점**
- AI 서버 비용 $0
- 오프라인에서도 동작
- 네트워크 지연 없음

**단점**
- 구현 난이도 높음
- 모델 배포 시 앱 업데이트 필요
- 배터리 소모 증가

현재 구조(30프레임 윈도우)를 폰에서 구현하려면 기술적으로 해결해야 할 부분이 있어서  
현재는 Option C를 채택하지 않고 서버 추론 방식을 유지하고 있습니다.
