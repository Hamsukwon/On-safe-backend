### v3.0 상세 변경 내역 (2026-05-13)

#### 🏗️ 아키텍처
- 이중 서버 구조 명시 — Python AI 서버(FastAPI)와 Kotlin 서버(Spring WebFlux) 역할 분리
- 응답 형식 분리 — Python: flat JSON / Kotlin: `{ success, message, data }` 래퍼
- 서버간 통신 흐름 추가 — `POST /internal/frame/{userId}` → Redis pub/sub → WebSocket

#### 🔢 수치 오류 수정
- `score` 범위: `0.0~1.0` → **`0~100`** (코드 실측값 반영: `predict_proba * 100`)
- 위험 레벨 임계값: `0.4/0.7` → **정상(0~50) / 주의(51~75) / 위험(76~100)** 으로 수정
- 낙상 로그 저장 조건: `fall == true` → **`score ≥ 76 OR fall == true`**

#### 🔐 인증·보안
- 로그인 응답에 **`access_token`, `refresh_token`, `token_type`** 추가 (v2.0: 누락)
- 로그인 요청에 **`device_id`** 필드 추가 (v2.0: 누락), 응답에도 echo 반환
- 로그아웃: v2.0의 "토큰 블랙리스트 등록" 오기재 수정 — 실제 컨트롤러는 단순 ok 반환
- **`POST /api/auth/refresh`** 신규 — `Refresh-Token` 헤더로 토큰 재발급

#### 📋 Auth 엔드포인트 변경
- **`POST /api/auth/check-id`** 신규 — 아이디 중복확인
- **`POST /api/auth/send-email-code`** 신규 — 회원가입 이메일 인증코드 발송 (TTL 3분)
- **`POST /api/auth/verify-email-code`** 신규 — 회원가입 이메일 인증코드 확인
- 비밀번호 재설정 단일 엔드포인트 → **3단계 분리**
  - `POST /api/auth/send-reset-code` (코드 발송, TTL 3분)
  - `POST /api/auth/verify-reset-code` (코드 검증)
  - `POST /api/auth/reset-password` (비밀번호 변경, 서버 측 단계 검증 없음)
- 아이디 찾기 응답: userId 뒷자리 `*` 마스킹 처리 추가 (`user001` → `use****`)
- 회원가입 요청: `address`, `address_detail` 선택 필드 추가

#### 📷 카메라·스트리밍
- **`PUT /api/camera/url/{device_id}`** 신규 — Kotlin 서버에서 카메라 URL 업데이트
- **`PUT /api/camera/session/{device_id}/start`** 신규 — 세션 시작 (Redis CONNECTING)
- **`PUT /api/camera/session/{device_id}/stop`** 신규 — 세션 종료 (Redis 삭제 + STOP 신호)
- **`GET /api/camera/session/{user_id}/status`** 신규 — 세션 상태 폴링
- **`WS /ws/camera/{user_id}`** 신규 — JPEG 프레임 실시간 수신 (`?token=` 인증)
- `PATCH /api/fall-logs/{userId}/{logId}/confirm` 담당 서버 수정: Python → **Kotlin** (경로도 수정)

#### 🗄️ DB 스키마
- `users` 컬렉션: `wardName` 필드 제거
- `realtime_data` 컬렉션 신규 — AI 추론 시 관절 피처 51개 + score 자동 저장, 사용자별 최신 2,000개 유지
- `fall_logs.score` 범위 표기 수정: `0.0~1.0` → `0~100`

#### 🔑 Redis 키
- Kotlin 관리 키와 Python 관리 키로 분리하여 명시
- 신규 키 추가: `email_verify:{mail}`, `reset_code:{userId}`, `camera:session:*`, `camera:frames:{userId}`
- v2.0에서 누락된 Python 내부 키 추가: `frame_count:{device_id}`, `frame:{device_id}`, `score:{user_id}`

#### 🔧 내부 API
- **`POST /internal/frame/{userId}`** 신규 — Python AI 서버 → Kotlin 서버 프레임 전달 (JWT 없음)
- **`POST /internal/realtime`** — Python 추론 후 Kotlin 호출 → `realtime_data/{userId}` 저장 (`GET /api/camera/score` 정상화)
- **`POST /internal/fall-log`** — 낙상 감지 시 Python이 Kotlin 호출 → `fall_logs` 저장 + FCM 발송 중앙화

#### 🐛 실측 테스트 버그 수정 (2026-05-13)
- **내부 API JSON 필드명 오류** — `/internal/realtime`, `/internal/fall-log` 요청 시 camelCase(`userId`, `deviceId` 등)로 전송하면 Kotlin SNAKE_CASE 정책에 의해 역직렬화 실패 → **snake_case(`user_id`, `device_id`, `log_id`, `is_confirmed`)로 수정**
- **`PUT /api/camera/url` 요청 필드명** — `cameraUrl` → **`camera_url`** (동일 SNAKE_CASE 정책)
- **`POST /api/devices/{userId}` upsert** — 다른 사용자 소유 기기 재등록 시 409 → 소유권 이전 후 **200 updated** 반환으로 변경
- **`realtime_data` Firestore 복합 인덱스** — `user_id ASC + timestamp DESC` 인덱스 생성 필요 (미생성 시 최신 2,000개 유지 쿼리 실패)
- **`GET /api/devices/{userId}` 빈 목록** — `.where().get()` → `.where().stream()` 교체로 Firestore async 쿼리 누락 수정

#### 📊 엔드포인트 수
- v2.0: 23개 (알림 엔드포인트 요약 테이블 누락 포함) → v3.0: **37개 REST + 1 WebSocket**

#### 📄 문서 구조
- Python AI 서버 구현 상세(처리 흐름, 모델 정보, realtime_data 필드)를 **부록 A**로 분리

## 🏗️ 서버 아키텍처

```
Android 클라이언트
    │
    ├─── REST ──────────────────→ Kotlin Spring Backend (포트 8080)
    │                             인증 · 유저 · 설정 · 낙상 로그
    │                             카메라 세션 · 알림
    │
    ├─── WebSocket ─────────────→ ws://server:8080/ws/camera/{userId}
    │                             실시간 카메라 프레임 수신
    │
    └─── REST ──────────────────→ Python AI Server (FastAPI)
                                  카메라 프레임 전송 + AI 추론
                                  기기 등록 · 조회

카메라 기기 (RTSP/HTTP)
    │
    └─── JPEG 프레임 ───────────→ Python AI Server
                                      │ POST /internal/frame/{userId}
                                      ↓
                                  Redis pub/sub (camera:frames:{userId})
                                      │
                                      └─────────────────────────────────→ Kotlin WebSocket Handler
                                                                           → 클라이언트에 Binary 전송
```

## 🔑 인증 방식

모든 보호 엔드포인트는 `Authorization: Bearer {accessToken}` 헤더가 필요합니다.

| 항목 | 값 |
|------|-----|
| 알고리즘 | HS256 |
| Access Token 만료 | 3,600초 (1시간) |
| Refresh Token 만료 | 604,800초 (7일) |
| 로그아웃 시 | Redis `bl:{token}` 블랙리스트 등록 (남은 TTL 적용) |

### Redis 키 구조

**Kotlin 서버 관리 키**

| 키 | 값 | TTL |
|----|-----|-----|
| `bl:{token}` | 로그아웃 블랙리스트 | 토큰 남은 만료 시간 |
| `email_verify:{mail}` | 회원가입 이메일 인증 코드 (6자리) | 3분 |
| `reset_code:{userId}` | 비밀번호 재설정 인증 코드 (6자리) | 3분 |
| `camera:session:{userId}:status` | `STANDBY` \| `CONNECTING` \| `LIVE` | 12시간 |
| `camera:session:{userId}:started_at` | ISO-8601 문자열 | 12시간 |
| `camera:frames:{userId}` | Redis pub/sub 채널 (영구 키 아님) | — |

**Python AI 서버 관리 키** (내부 전용 — 상세는 [부록 A](#부록-a-python-ai-서버-구현-상세) 참고)

| 키 | 값 | TTL |
|----|-----|-----|
| `score:{user_id}` | `{"score": float, "level": string}` JSON | 30초 |
| `frame_count:{device_id}` | 프레임 카운터 (int) | 10초 |
| `frame:{device_id}` | JPEG bytes (최신 프레임) | 5초 |

### 비밀번호 재설정 — 3단계 플로우

```
POST /api/auth/send-reset-code   → 인증코드 이메일 발송
          ↓
POST /api/auth/verify-reset-code → 코드 검증 (Redis 키 삭제)
          ↓
POST /api/auth/reset-password    → 비밀번호 변경
```

> ⚠️ 서버는 3단계 호출 시 2단계 완료 여부를 별도로 검증하지 않습니다.

#### `POST /api/auth/send-reset-code` — 1단계: 인증코드 발송

Redis에 `reset_code:{user_id}` 키로 6자리 코드 저장 (TTL 3분), 등록된 이메일로 발송.

**Request Body**

| 필드 | 타입 | 필수 | 설명 |

## 📡 4. 실시간 카메라 스트리밍 (WebSocket) — Kotlin 서버 ✨ 신규

### `WebSocket ws://server:8080/ws/camera/{user_id}`

**인증:** 쿼리 파라미터 `?token={accessToken}` (JWT 검증 — userId 불일치 시 403)

**메시지 형식:** Binary (JPEG bytes)

**동작:**
```
1. 클라이언트 → WebSocket 연결
2. 서버 → Redis 채널 camera:frames:{userId} 구독
3. Python AI 서버가 Redis에 Base64 프레임 발행
4. Kotlin 서버 → Base64 디코딩 → 바이너리 메시지로 클라이언트 전달
5. 첫 프레임 수신 시 세션 상태 자동으로 LIVE 전환
6. PUT .../stop 수신 시 연결 종료 (takeUntilOther)
7. 연결 종료 시 세션 상태 STANDBY 복구
```

**주의사항:**
- 서버 측 재연결 없음 — 클라이언트가 재연결 로직 구현 필요
- Base64 인코딩은 Spring Data Redis 3.4.x String 제약으로 필수
- 에뮬레이터에서 PC localhost 접근: `ws://10.0.2.2:8080/ws/camera/{userId}`

## 🔬 부록 A. Python AI 서버 구현 상세

> 앱 개발에는 불필요한 서버 내부 구현 정보입니다.  
> 파일 위치: `app/domain/camera/service.py`, `app/ai/engine.py`, `app/ai/buffer.py`

---

### A-1. `POST /api/camera/stream` 처리 흐름

```
1.  JPEG bytes → Redis frame:{device_id} 저장 (TTL 5초)
2.  frame_count:{device_id} 카운터 +1 (TTL 10초)
3.  카운터 < 30이면 {"score": 0.0, "fall": false} 즉시 반환 (버퍼 미달)
4.  MediaPipe Pose → 33개 관절 (x, y, z) 좌표 추출
5.  Savitzky-Golay 필터로 좌표 노이즈 제거 (window=5, poly=2)
6.  무게중심(hip 중점) 기준 중심화 → hip 간 거리로 스케일 정규화
7.  15개 관절 × 3 (각도·각속도·각가속도) = 45개 피처 계산
8.  무게중심 동역학 6개 피처 계산 (speed, acceleration, displacement 등)
9.  Decision Tree 모델 + StandardScaler → score(0~100), fall(bool) 추론
10. realtime_data Firestore 저장 (51개 피처 + risk_score + user_id + timestamp) — ML 분석용 auto-ID 문서
11. POST /internal/realtime → Kotlin이 realtime_data/{userId} 갱신 (GET /api/camera/score 정상화)
12. score ≥ 76 OR fall == True → POST /internal/fall-log → Kotlin이 fall_logs 저장 + FCM 발송
```

---

### A-2. Redis 내부 키 상세

| 키 | 용도 | TTL | 비고 |
|----|------|-----|------|
| `frame_count:{device_id}` | AI 추론 트리거 카운터 | 10초 | 30 이상이면 추론 실행 |
| `frame:{device_id}` | 최신 JPEG 프레임 bytes | 5초 | `save_latest_frame()` |
| `score:{user_id}` | 최신 추론 결과 캐시 | 30초 | `GET /api/camera/score` 에서 읽음 |

### A-4. AI 모델 정보

| 항목 | 값 |
|------|-----|
| 포즈 추출 | MediaPipe Pose (`model_complexity=1`) |
| 분류 모델 | Decision Tree (`pkl/decision_tree_model.pkl`) |
| 스케일러 | StandardScaler (`pkl/scaler.pkl`) |
| 노이즈 필터 | Savitzky-Golay (window=5, poly=2) |
| 추론 방식 | `predict_proba()[1] × 100` → score, `predict()` → fall |
| 비동기 처리 | `loop.run_in_executor()` (스레드 풀) |
